  1. 把前 49 轮关于状态、组织、自约束、复利、生命周期的局部 hardening 收束成单一 canonical operating model。

## Divergent Hypotheses

1. `继续只优化 Hosted Kernel 的 install/update surface`
   - 入口更稳，但仍然没有回答“大任务自治”和“组织化协作”。
2. `做字面上的公司模拟`
   - 用更多部门、会议、agent 称号来制造组织感。
   - 风险是形式越来越厚，执行协议仍然很软。
3. `做 contract-first self-governing harness`
   - 保留低侵入 Hosted Kernel
   - 但把 repo-local state、ownership、handoff、review、compounding、lifecycle 全部合同化
   - 让 agent 不是“在聊天里像公司”，而是在协议里像公司

## First Principles Deconstruction

1. 真正限制 agent 独立完成大任务的，不是模型智力，而是：
   - 状态真相不清
   - ownership 不清
   - 交接不清
   - stop conditions 不清
   - 失败后的恢复路径不清
2. “像公司一样组织”如果不能落到：
   - 谁拥有哪类产物
   - 谁能推进哪些状态
   - 谁负责 review 与 acceptance
   - 哪些动作必须停线
   那它就只是组织叙事，不是操作系统。
3. “及时复利”不是多写文档，而是：
   - 高价值输出被捕获
   - 被压缩成可复用约束
   - 被下一轮 agent 真正读取和继承
4. 因此最佳方向不是“更多角色 + 更多文档”，而是：
   - 更硬的 source state
   - 更清晰的组织合同
   - 更强的 fail-closed 自约束
   - 更快的学习闭环

## Convergence To Excellence

采纳第 3 条路线，正式定义：

`Self-Governing Agent Company Harness v2.50`

它不是对 `Hosted Kernel v1.31` 的推翻，而是它的上层 operating model：

1. `v1.31` 回答：低侵入 consumer repo 如何被 Hosted Kernel 稳定接管
2. `v2.50` 回答：被接管之后，agent 如何在 repo-local state 与硬合同下，组织化地完成大任务并复利

## Canonical Objective

系统目标固定为：

1. agent 能独立推进 major task
2. agent 之间能安全并行、清晰交接、稳定收敛
3. 关键动作都必须经过 plan/apply、review、doctor、repair、stop-the-line 等合同约束
4. 决策、brief、source note、incident、retro、release learning 必须形成复利闭环
5. 所有这些能力必须可通过 fixture 和 acceptance suite 自证

## Canonical Layering

### 1. Hosted Kernel Layer

沿用 `v1.31`：

1. root `AGENTS.md / CLAUDE.md / GEMINI.md`
2. `.harness/entrypoint.md`
3. `.harness/kernel-dispatch.toml`
4. provider-native carrier
5. `.harness/workspace/current/project-context.md`

职责：

1. 提供 deterministic first hop
2. 锁定 carrier identity、compatibility、managed surface
3. 保持 consumer repo 低侵入

### 2. Repo-Local State Layer

这是 `v2.50` 的新增重心。

`.harness/workspace/` 里的状态必须拆成三层：

1. `source state`
   - `command-journal`
   - `work-items`
2. `projections`
   - `boards`
   - `progress`
   - `status snapshots`
3. `compounding assets`
   - `decisions`
   - `briefs`
   - `source-notes`
   - `incidents`
   - `retros`
   - `playbook-deltas`

核心原则：

1. 只能改 source，不能改 view
2. append-only 优先，覆盖写是例外
3. 任何 projection 都必须能从 source rebuild / replay

### 3. Organization Layer

组织不是“部门故事”，而是责任分配合同。

角色族固定为：

1. `principal`
2. `manager`
3. `lead`
4. `worker`
5. `reviewer`

每类角色只承担有限职责集合。

关键原则：

1. manager 负责收敛，不默认拥有执行权
2. worker 负责窄职责产出，不默认拥有 closure 权
3. reviewer 独立于 implementation lane
4. integration ownership 永远留在主线程或明确 owner

### 4. Autonomy Layer

自治必须被 gate ladder 约束。

四级 gate：

1. `safe-read`
2. `planned-write`
3. `reviewed-apply`
4. `escalated-stop`

关键原则：

1. 高风险动作必须 plan/apply 分离
2. claim 没有 evidence 只能算 hypothesis
3. stop-the-line 比“尽快推进”优先级更高

### 5. Compounding Layer

复利不是 archive，而是 capture -> compress -> reuse。

canonical asset classes：

1. `decision`
2. `brief`
3. `source-note`
4. `incident`
5. `retro`
6. `playbook-delta`

关键原则：

1. 只沉淀高价值输出
2. source notes 必须有 freshness
3. retro 必须回写 action
4. release learnings 必须回流系统规则

### 6. Lifecycle Layer

全生命周期命令必须统一成：

1. `detect`
2. `survey`
3. `plan`
4. `apply`
5. `doctor`
6. `repair`
7. `release / promote`

关键原则：

1. 没有 survey 的 apply 不可信
2. 没有 doctor green 的升级不可信
3. 没有 fixture 的 lifecycle 不可发布

## State Management Contract

### Source State

`command-journal` 是 append-only mutation ledger。

每条 journal 至少记录：

1. `actor`
2. `command`
3. `target`
4. `idempotency_key`
5. `intent`
6. `timestamp`
7. `result_state`

`work-item` 是带 acceptance 合同的执行对象，至少包含：

1. `intent`
2. `owner`
3. `delegated_to`
4. `status`
5. `acceptance`
6. `evidence`
7. `blockers`
8. `next_action`

### Mutation Rules

1. mutating command 必须有 `idempotency_key`
2. 大任务状态推进必须带 ownership stamps
3. 并行写必须通过 `lease`
4. owner 变更必须单独记账

### Replayability

1. snapshot 是 source state 的 point-in-time materialization
2. replay 是从 journal 重建到目标状态
3. 不能 replay 的状态，不算 canonical state

### State Acceptance

最小 suite 包括：

1. fresh state
2. duplicate command
3. lease conflict
4. replay rebuild
5. projection drift

## Multi-Agent Organization Contract

### Responsibility Matrix

采用 `RACI-lite`：

1. `request`
2. `own`
3. `execute`
4. `review`
5. `approve`

最小要求：

1. major task 必须至少有 owner、executor、reviewer
2. manager 不默认兼任 reviewer
3. worker 不得自我验收 major task

### Task Decomposition

每个 major task 必须生成 decomposition tree。

树节点固定分为：

1. `analysis`
2. `implementation`
3. `verification`
4. `integration`

原则：

1. 大任务先分解，再委派
2. 父任务只能在子任务 acceptance 满足后推进

### Work Package

每个可委派 work package 至少必须包含：

1. `scope`
2. `owned_paths`
3. `expected_output`
4. `verification`
5. `stop_conditions`

没有 `owned_paths` 的 package 不允许并行委派。

### Handoff

handoff 不靠对话，靠 artifact。

artifact types 固定为：

1. `finding`
2. `patch`
3. `decision-request`

handoff contract 必须包含：

1. `task_id`
2. `current_owner`
3. `next_owner`
4. `input_artifact`
5. `expected_output`
6. `stop_condition`
7. `return_path`

### Review Loop

review lane 独立于 work lane。

manager review loop 输出固定为：

1. `accept`
2. `revise`
3. `split-further`
4. `stop`

### Escalation

只有四类升级：

1. `schema ambiguity`
2. `ownership conflict`
3. `unsafe mutation`
4. `missing evidence`

不在矩阵里的问题默认 local resolve。

## Self-Constraint And Autonomy Contract

### Gate Ladder

1. `safe-read`
2. `planned-write`
3. `reviewed-apply`
4. `escalated-stop`

不同动作必须落在对应 gate。

### Plan / Apply

所有高风险写动作必须分成：

1. `plan`
2. `apply`

`plan` 必须描述：

1. intended files
2. state targets
3. evidence
4. rollback hooks

### Self-Checklist

analysis、implementation、review 都必须执行最小自检。

执行者必须显式记录：

1. 已覆盖项
2. 未覆盖项
3. residual risk

### Evidence

只接受四类 evidence：

1. `test/check output`
2. `file diff`
3. `cited docs`
4. `reproduced behavior`

### Stop-The-Line

stop triggers 固定为：

1. schema ambiguity
2. unsafe overwrite
3. conflicting source of truth
4. verification failure
5. provider mismatch

触发后只能：

1. `doctor`
2. `repair`
3. `escalate`

### Recovery Tree

恢复路径固定为：

1. `doctor`
2. `repair`
3. `replay`
4. `escalate`

repair 只允许修：

1. generated metadata
2. managed blocks
3. projection rebuild

不得改：

1. append-only assets
2. 用户正文

### Budget And Timebox

analysis、implementation、research 都必须有默认预算。

超预算只能：

1. `split`
2. `escalate`

## Timely Compounding Contract

### Asset Taxonomy

复利资产固定为：

1. `decision`
2. `brief`
3. `source-note`
4. `incident`
5. `retro`
6. `playbook-delta`

### Decision Quality

decision entry 最少必须包含：

1. `decision`
2. `why`
3. `alternatives rejected`
4. `implications`
5. `next check date`

并且必须能回指到：

1. work item
2. evidence

### Brief Lifecycle

固定为：

1. `draft`
2. `decision-pack`
3. `accepted`
4. `archived`

brief 是转换层，不是 backlog 垃圾桶。

### Source Freshness

source-note 至少包含：

1. `source`
2. `captured_at`
3. `freshness_window`
4. `claim_scope`

超过 freshness window 的外部事实默认降级。

### Cadence

最小 cadence：

1. `daily digest`
2. `weekly retro`

digest 汇总新增事实；retro 总结模式与系统改进。

### Incident And Release Learning

incident 必须记录：

1. failure mode
2. blast radius
3. recovery path

每个 release channel 都必须回写：

1. what worked
2. what failed
3. what to harden next

### Archive Policy

archive 分三层：

1. `active`
2. `warm`
3. `cold`

原则：

1. 保留历史
2. 不保留热噪音

## Workflow And Lifecycle Contract

### Bootstrap-State Detection

repo 状态固定分为：

1. `fresh`
2. `partial`
3. `adopted`
4. `drifted`
5. `broken`
6. `dogfood`

所有 lifecycle 命令先判断状态，再决定 allowed next action。

### Lifecycle Phases

bootstrap、adopt、update、repair 全部采用：

1. `survey`
2. `plan`
3. `apply`

### Profiles

profile 是 first-class object，支持：

1. `base`
2. `overlays`

profile 只影响：

1. seed
2. thresholds
3. cadence

不改变核心 contracts。

### Local Overrides

local-overrides 分三层：

1. `provider-overrides`
2. `project-local-not-adopted`
3. `operator-local`

它们是合法出口，不是第二套真相。

### Entry Mirrors

root `AGENTS / CLAUDE / GEMINI` 是唯一 root 侵入面，必须：

1. mirror parity
2. overlay prelude 带 contract version
3. drift 直接触发 doctor failure

### Channels

release channels 固定为：

1. `dogfood`
2. `canary`
3. `stable`

consumer repo 默认只允许 stable。

## Acceptance And Fixture Contract

### Acceptance Suites

系统至少有五套 suite：

1. `state`
2. `organization`
3. `autonomy`
4. `compounding`
5. `workflow`

### Fixture Matrix

最小 fixture matrix：

1. fresh repo
2. legacy agent repo
3. mixed sections
4. provider-heavy repo
5. monorepo
6. drifted repo
7. broken repo

### CI Gate

最小 gate：

1. doctor strict
2. suite smoke
3. bootstrap replay
4. update-plan smoke

没有 fixture delta 的 contract 变更，不允许进入发布。

## Product Boundary

固定边界：

1. `harness-framework = distribution product repo`
2. `consumer repo = runtime state + root overlay + repo-local projections`
3. `trading-agent = dogfood/canary consumer`

不允许把 framework source of truth 与 dogfood instance 混成一个仓的同一层。

## What This Version Actually Locks

`v2.50` 锁定的不是某一个脚本或某一个目录，而是以下 operating model：

1. Hosted Kernel 继续保留低侵入分发
2. repo-local first hop 继续保持 deterministic
3. 状态管理必须进入 source/projection/compounding 三层
4. 多 agent 组织必须靠 ownership、handoff、review、escalation 合同运行
5. 自治必须被 plan/apply、doctor、repair、stop-the-line、evidence、budget 约束
6. 决策、brief、source-note、incident、retro、release learning 必须形成及时复利闭环
7. 所有这些都必须能被 fixture 和 acceptance suite 自证
